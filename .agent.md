# Project Agent Guide

## Mission
Build and maintain a WordPress website with a custom chatbot plugin that supports both public and private content, is secure by default, and remains portable to a fallback off-the-shelf plugin if needed.

## Primary strategy
- Default to the custom plugin path.
- Preserve a fallback path to existing AI plugins.
- Do not couple UI, data, or permissions to one runtime.

## Ownership
- Sasha owns UI/UX, visual system, page structure, Divi presentation.
- Pawel owns plugin architecture, AI/chatbot logic, access control, telemetry, security sign-off.
- Shared decisions: content classification, merge readiness, release acceptance.

## Core architecture rules
1. WordPress/Divi is the presentation shell.
2. Chatbot logic lives in the custom plugin, not in theme templates.
3. Public and private AI knowledge must be explicitly classified.
4. Only approved content may be used by the AI.
5. The runtime must support browser-based local testing.
6. The UI contract must remain stable so final designs can replace the mock UI without backend rewrites.
7. All operationally important settings must be documented in the canonical config inventory.

## Local development rules
- Maintain a local WordPress instance with the plugin enabled.
- Maintain test users and roles for public and authenticated scenarios.
- Maintain deterministic smoke-test prompts.
- Ensure the chatbot can be exercised in an Antigravity-enabled Chrome browser.
- Prefer reproducible local URLs and predictable seeded data.

## Security rules
- Use capability checks for all privileged actions.
- Protect state-changing requests with nonces.
- Validate input, sanitize input, escape output.
- Never expose secrets in frontend or theme code.
- Keep admin and public code paths separate.
- Log permission failures, retrieval failures, and unexpected runtime errors.

## Documentation rules
Update docs whenever changing:
- plugin inventory
- membership settings
- chatbot/AI configuration
- analytics setup
- forms
- redirects
- roles/permissions
- environment/security settings

## UI/UX rules
- Respect Sasha’s final component and state definitions.
- Support idle, loading, streaming, answer, auth-required, error, and escalation states.
- Do not hardcode styling assumptions that prevent later theme replacement.

## Data and permission rules
- AI may only use content that is both approved and within the current user’s visibility scope.
- Do not infer access from page context alone.
- Treat private content as deny-by-default.

## Testing rules
Before marking work complete, verify:
- public page behavior
- authenticated private behavior
- role changes
- denied access paths
- browser console/network cleanliness
- feedback/analytics event flow
- regression against smoke-test prompts

## Fallback rules
Trigger fallback planning if:
- custom plugin threatens MVP timeline
- access control is not trustworthy
- local browser workflow is unstable
- admin/config burden grows too high

Fallback implementations must preserve:
- the same content visibility model
- the same documentation inventory
- the same UI contract
- the same telemetry expectations

## Output expectations for agents
When proposing changes, include:
- system design impact
- interface/data contract impact
- security implications
- migration impact
- test cases
- docs that must be updated

Do not submit work that changes runtime behavior without also updating the relevant docs.
